# Protocol Core Coverage

This matrix tracks the TypeScript protocol-core coverage against the `wot-spec` conformance manifest and test vectors.

Legend:

- Full: implemented and checked by `packages/wot-core/tests/ProtocolInterop.test.ts`
- Partial: protocol pieces are covered, but the full profile behavior is not implemented in TS
- External: intentionally validated outside TS protocol-core
- Not covered: no TS protocol-core implementation yet
- Prose-backed coverage: no dedicated test vectors exist; behavior is implemented and tested against the normative prose of the cited spec sections

## Schema Coverage

JSON Schema validation generally remains owned by `wot-spec`:

```bash
npm run validate:schemas
```

The TypeScript protocol-core validates protocol behavior against vectors and now mirrors focused Sync 002/003/005 schema constraints for log-entry payloads, plaintext-envelope shapes, membership message UUID-v4 identifier and `spaceId` fields, and the `space_capability_jws` capability payload invariants needed before signing and after JWS decoding. Complete schema-suite validation remains centralized in `wot-spec`.

## Current Gaps

- Complete SD-JWT VC implementation beyond the current trust-list vector requirements, including holder binding / `cnf` verification.
- Trust 002 delivery, transport envelopes, wallet persistence, contact updates, profile publication, active QR challenge persistence, and persistent nonce/pending-counter state remain outside protocol-core coverage. Application coverage creates canonical lowercase UUID-v4 QR challenge nonces, signs nonce-bound VC-JWS artifacts, covers the `trust002_jti_nonce_binding` parser behavior after PR #79 lands, and exercises in-memory active-challenge and pending-counter decisions only.
- HMC trust-score path aggregation, hop-limit propagation, minimum-score policy, and anti-gaming rules; these remain deferred to real-life-org/wot-spec#9.
- HMC H03 `trust-list-delta` authority and forwarding semantics are not protocol-core behavior in this slice: no envelope `from`/`to` binding to SD-JWT issuer/audience/holder, hop-limit forwarding, Sent-Log, replay cache, storage, or SD-JWT disclosure verification. These remain tracked in `real-life-org/wot-spec#43`.
- HMC Gossip beyond the H03 envelope slice, Payment (H02), Sent-Log behavior, piggybacking, Sync inbox forwarding, RLS display fields, and application workflow behavior.
- HMC H03 `trust-list-delta.body.delta` currently mirrors the schema regex exactly, including empty disclosure segments. Whether those segments should be invalid SD-JWT-VC syntax remains tracked in `real-life-org/wot-spec#44`.
- JSON Schema validation in TS; currently intentionally centralized in `wot-spec`.
- Complete broker Challenge-Response runtime authentication remains out of protocol-core scope. Protocol-core now covers deterministic 32-byte Base64URL nonce formatting/parsing, consumed-nonce replay classification, Broker-Auth-Transcript JCS signing bytes, exact pending-challenge `did`/`deviceId`/`nonce` binding, `challenge-response` Control-Frame wire-level `signature` parsing/serialization, and Ed25519 signature verification with caller-supplied public key bytes. DID resolution, key-discovery policy, WebSocket connection binding storage, consumed-nonce persistence, and broker runtime behavior remain outside these helpers.
- `real-life-org/wot-spec#16` is closed: `wot-identity@0.1` owns JCS/JWS base rules, concrete JWS artifact vectors are owned by the profile that defines the artifact, and AES/ECIES vectors belong to `wot-sync@0.1`. `real-life-org/wot-spec#17` is closed: `wot-identity@0.1` owns the `jcs_canonicalization` primitive vector section, including number formatting and invalid JSON-number handling.
- Sync 004 profile-service resource-specific JWS `typ`, dedicated schema ownership for profile-service resources, top-level/profile-object additional-property/key-material ownership, and vector ownership are deferred pending `real-life-org/wot-spec#34`.
- The signed Sync 003 `device-revoke` Broker Control-Frame wire/JWS shape is covered by protocol-core against the `broker_device_revoke_control_frame` vector. Real broker WebSocket connection binding, registration handshake state, routing, persistence, retry behavior, device-list mutation, inbox deletion, DID resolution policy beyond caller-supplied `did:key` key material, and runtime error emission remain outside protocol-core.
- Device-delegation revocation distribution, device-admin delegation chains, broker-auth challenge handling, delegated log-entry signing, Sync integration, and application device-management APIs remain out of protocol-core scope for this slice.

## External Boundaries

- DIDComm plaintext-envelope library compatibility is a transport boundary. TypeScript exposes pure shape helpers for the Sync 003 plaintext envelope, while DIDComm parser compatibility remains validated in `wot-spec` against DIDComm libraries.
