# Slice R — Relay durable-log (Spaces-Verlust-Fix)

**Status:** implemented on `feat/slice-r-relay-durable-log`, 78/78 relay tests green, typecheck clean (incl. VE-3a author-binding). Gated on the Sync-002 spike GO (`packages/sync-spike`). Sister-slice **A** (client log-write + sync-request/apply) depends on this (`dependsOn: R`); A+R merge together after an end-to-end cold-reconstruction test on the real Yjs adapter.

## What changed

The relay was a transient delivery queue (`queue.ts`: `queued → delivered → ACK → DELETE`); space content lived only in browser caches, so all-caches-empty meant **permanent data loss**. Slice R adds a **durable, retained, append-only per-doc log** as the source of truth and a **sync-request → sync-response catch-up server**, while leaving the inbox queue (the 4 encrypted inbox types + `ack/1.0` semantics) **unchanged**.

- `src/log-store.ts` (new) — `DocLog` over better-sqlite3: `doc_log(doc_id, device_id, seq, content_hash, entry_jws, created_at, PK(doc_id,device_id,seq))`. `appendEntry` (idempotent via `INSERT OR IGNORE`), `getContentHash`, `getSince`/`getSinceWithTruncation` (by heads, deterministic `device_id,seq` order, limit/truncation), `getHeads`, stats. All hashing lives here (relay.ts stays crypto-free for the source guard). Never decrypts `data`.
- `relay.ts` — `handleSend` gained two sibling branches before generic routing (mirroring the `ack/1.0` branch). Only a **structurally valid** `log-entry/1.0` (`parseLogEntryMessage`) or `sync-request/1.0` (`parseSyncRequestMessage`) is diverted; a merely-typed-but-malformed envelope **falls through** to the unchanged inbox routing (this is what keeps the regression byte-for-byte intact). Ingest verifies the JWS (`verifyLogEntryJws`), reads `docId/deviceId/seq` from the **authenticated payload** (authority via `authorKid`, Sync 002 Z.126), runs `classifyBrokerSeqCollision`, then appends + live-relays to connected `envelope.to` recipients (sender socket excluded). `sync-request` is served only to the requesting authenticated socket; empty heads ⇒ full log (cold reconstruction).
- Shared SQLite handle: `RelayServer` owns one connection and passes it to both `OfflineQueue` and `DocLog` (borrowed handle → `close()` no-op). Avoids the `:memory:` split-DB problem; prod uses one file.
- `start.ts` — clean shutdown on **SIGTERM** as well as SIGINT; a defensive `unhandledRejection` logger.

## What is validated

- **Cold reconstruction (teeth):** producers push real log-entry JWS then **disconnect**; a fresh device connects *after* and rebuilds the full doc via `sync-request` with empty heads — reconstruction is served from the durable log, not live broadcast (`DurableLogSync.test.ts`).
- **Nonce safety (Sync 002 Z.81):** a divergent entry at an existing `(docId,deviceId,seq)` is rejected `SEQ_COLLISION_DETECTED` + `restore-clone-required` **before** store and before relay; idempotent retransmit neither re-stores nor re-broadcasts. (Adversarially probed with concurrent + forced-window races — invariant holds; the author-binding + collision check + insert are one synchronous SQLite transaction, plus `INSERT OR IGNORE` backstop.)
- **Author-binding (VE-3a):** the first `authorKid` to write a `(docId,deviceId)` owns that seq/nonce namespace (`doc_device_author` table); a foreign author — even one signing a perfectly valid JWS with its own `authorKid` — is rejected `AUTHOR_MISMATCH` **before** store and before relay, at both a new seq and the owner's existing seq. Owner lookup/insert + collision check + append run in **one SQLite transaction inside `appendEntry`** (no pre-check-then-append), so first-writer-wins is race-free (better-sqlite3 is synchronous; the transaction adds crash-atomic rollback). Closes the squat / write-DoS / log-poisoning finding from the review.
- **Regression:** the 4 inbox types + `ack/1.0` queue semantics are byte-for-byte unchanged (pre-existing test files have zero diff vs HEAD; the divert-guard only intercepts structurally-valid log/sync envelopes).
- **Robustness:** a transient SQLite read/write failure during ingest or sync-request now surfaces as an `INTERNAL_ERROR` frame to the sender and **does not crash the relay** (previously: unhandled rejection → process exit on Node 22, sender hung). Locked by two tests.
- Relay stays **blind** (ciphertext + metadata only); source guard green for the right reason; no new deps; no new `#sig-0`/issuer hardcoding.

## Known limitations / deferred (flagged for follow-up)

1. **`deviceId` ↔ `authorKid` squat — RESOLVED in R (VE-3a).** The original review found that the `(docId,deviceId,seq)` collision key excludes `authorKid`, letting a validly-signing non-owner squat another device's slot (write-DoS + reconstruction poisoning). VE-3a closes this with first-writer-wins author-binding (see "What is validated"). **Residual, deferred:** a brand-new (never-seen) `deviceId` minted by *any* authenticated party can still open a fresh namespace with garbage — because the relay does not gate ingest on space membership (any `did:key` may connect). That is the **capability-gated ingest** work below (item 2), **not** VE-3a. Full cryptographic `deviceId`↔device-key proof remains Identity-004 / Phase 2.
2. **No membership / capability authz on ingest + `sync-request` — NEXT Sync-003 slice (high priority, before the A+R merge gate).** Any authenticated device can write a fresh `(docId,deviceId)` namespace and can pull the full retained **ciphertext** log + per-device head metadata of any `docId` it knows. Confidentiality holds (E2E ciphertext), but `deviceId`s, seq counts, `authorKid`s and timestamps are plaintext JWS metadata, and the log is open to garbage under fresh deviceIds. Per the updated directive this is the **next slice** (membership/authz + capability lifecycle), prioritised post-R but **before A+R merge**, so the durable log is both squat-robust (VE-3a) and write-gated.
3. **Live broadcast routes on the unverified transport `envelope.to`** (not payload-derived recipients) — consistent with the relay's existing routing model; ciphertext-only, so impact is unsolicited delivery / metadata.
4. **Log-entry receipt status is `delivered` even with no live recipient connected.** For the durable-log model, durable persistence is the success condition (the entry will be served on catch-up), so `delivered` means "durably accepted into the log", not "received by a live socket" — a semantic divergence from the inbox path's `accepted`/`delivered`. Slice A defines what it needs from the receipt.
5. **No pruning / snapshots** — the log grows monotonically by design (Slice C handles compaction).
