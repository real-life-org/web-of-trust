# WoT Trust Conformance Notes

This document tracks reference-implementation coverage for `wot-trust@0.1`. It is non-normative; `../wot-spec/02-wot-trust/` remains the authority.

## Trust 002 Online Verification

- **Spec refs:** `../wot-spec/02-wot-trust/002-verifikation.md`, `../wot-spec/schemas/qr-challenge.schema.json`, and the QR challenge valid/invalid schema examples.
- **Implementation:** `packages/wot-core/src/protocol/trust/qr-challenge.ts` owns raw QR parsing and already-verified Verification-Attestation acceptance decisions. `packages/wot-core/src/application/verification/verification-workflow.ts` owns application state: QR challenge creation, deterministic VC-JWS Verification-Attestation creation, active challenge tracking, consumed nonce history, and pending counter-verification state. `packages/wot-core/src/ports/VerificationStateStore.ts` defines the async durable boundary for consumed nonce and pending counter-verification records; `packages/wot-core/src/adapters/storage/InMemoryVerificationStateStore.ts` provides the volatile reference/test implementation.
- **Tests:** `packages/wot-core/tests/ProtocolInterop.test.ts` covers protocol parsing and acceptance decisions, including canonical lowercase UUID-v4 QR challenge nonce enforcement and the `trust002_jti_nonce_binding` vector once PR #79 lands. `packages/wot-core/tests/VerificationWorkflow.test.ts` covers application QR creation, rejection of injected non-canonical/non-v4 QR nonces before storing an active challenge, optional broker serialization, active challenge reset, nonce consumption, replay rejection, 24-hour retention boundaries, expired challenge rejection, restart-safe remote/unbound classification, nonce-bound VC-JWS Verification-Attestation creation, signed `inResponseTo` preservation, `inResponseTo`-bound counter-verification acceptance, store-backed consumed nonce replay rejection across workflow instances, store-backed pending counter acceptance across workflow instances, and expired pending counter rejection through the store path. `packages/wot-core/tests/VerificationStateStore.test.ts` covers the reference in-memory port implementation.
- **Mutual in-person flow:** one side creates a nonce-bound Verification-Attestation whose `id`/`jti` is `urn:uuid:<normalized-challenge-nonce>`, stores local pending counter state keyed by that `jti`, and sends the VC-JWS through an out-of-scope delivery path. The challenge owner accepts the already verified payload as `accept-in-person` only when the active nonce matches the full-string `jti` grammar, then can answer with a signed Counter-Verification-Attestation whose top-level `inResponseTo` exactly matches the original `jti`; only then can the response become `accept-mutual-in-person`.
- **Creation path:** `VerificationWorkflow.createVerificationAttestation` and `createCounterVerificationAttestation` are the Trust 002 VC-JWS reference path. Verification-Attestation creation emits both VC `id` and JWT `jti` as `urn:uuid:<normalized-challenge-nonce>`. Counter-Verification-Attestations use their own `jti` and bind back through top-level `inResponseTo`. Legacy `Verification` proof helpers remain source-compatible for older callers, but they are not the Trust 002 VC-JWS conformance path.
- **Parser path:** PR #79 replaces broad substring matching with the Trust 002 `trust002_jti_nonce_binding` vector behavior. The parser full-matches `urn:uuid:<uuid>`, normalizes only the UUID portion for active challenge comparison and nonce-history replay classification, treats multi-UUID, prefix/suffix, wrong-delimiter, wrong-namespace, and other non-canonical forms as unbound remote verifications, and leaves non-canonical `urn:uuid:` prefix case handling unbound pending `wot-spec#63`.
- **Boundaries:** The creation helpers only construct and sign VC-JWS artifacts through the Trust 001 / Identity 002 protocol helper. Acceptance methods consume already verified `AttestationVcPayload` values. This slice does not send messages, persist wallet records, update contacts, publish discovery records, persist active QR challenges, create transport envelopes, or define delivery/ACK behavior. Consumed nonce history and pending counter-verification state can be routed through an injected `VerificationStateStore`; without one, the workflow preserves the previous in-memory behavior.
- **Persistence caveat:** `activeQrChallenge` remains intentionally in-memory and is not part of `VerificationStateStore`; a restart clears active QR challenge state and should generate a fresh challenge. Consumed nonce history and pending counter-verification state now have an injected durable port boundary, while the bundled in-memory implementation remains volatile.
- **Open questions:** Spec PR #62 resolved the Trust 002 full-match `jti` grammar previously tracked under `wot-spec#47`; PR #65 resolved `wot-spec#38` by requiring Trust QR challenge nonces to be canonical lowercase UUID v4 values. `wot-spec#63` now tracks only whether the literal `urn:uuid:` prefix is case-sensitive. Until that is resolved, non-canonical prefixes are treated as unbound remote verifications rather than nonce-bound in-person proof.

## Trust 001 No attestation-ack semantics

- **Spec refs:** `../wot-spec/02-wot-trust/001-attestations.md` (Z.147, Z.149) and `../wot-spec/CONFORMANCE.md` (Z.69).
- **Normative rule:** `wot-trust@0.1` defines no `attestation-ack` and no semantic acceptance confirmation. Attestations are gifts: after delivery they belong to the holder (001 Z.147). Transport ACKs from the sync layer only confirm processing or durable buffering of an inbox message for a specific device; they MUST NOT be interpreted as an acceptance or trust signal for an attestation (001 Z.149, CONFORMANCE Z.69). A later publication in the recipient's profile is the recipient's deliberate public feedback.
- **Implementation:** The reference implementation carries no `attestation-ack` message type and no `acknowledged` delivery/receipt status. Delivery tracking in the demo (`apps/demo/src/services/AttestationService.ts`) maps only transport states (`sending`/`queued`/`delivered`/`failed`) from relay `onReceipt` (Sync 003 `ack/1.0`). Publish-consent is tracked separately via the `accepted` flag (`setAttestationAccepted`), which gates profile publication and is not a wire-trust acknowledgement.
- **Boundaries:** This document does not normatively define delivery/ACK behavior; see `../wot-spec/02-wot-trust/` for the authority and `packages/wot-core/src/protocol/COVERAGE.md` for the protocol-core scope note (`attestation-ack` is not a protocol concept — Trust 001 Z.147 + CONFORMANCE Z.69; resolved in slice 1.B.2).

## Trust 001 Timestamp Precision

- **Spec refs:** `../wot-spec/02-wot-trust/001-attestations.md` and `../wot-spec/schemas/attestation-vc-payload.schema.json`.
- **Implementation:** `packages/wot-core/src/protocol/trust/attestation-vc-jws.ts` validates `validFrom` and optional `validUntil` as whole-second RFC3339 date-time strings with explicit timezone, accepts lowercase RFC3339 separators, rejects fractional seconds including `.000Z`, and compares normalized instants exactly against integer `nbf` and optional `exp`. `AttestationWorkflow` and `VerificationWorkflow` emit canonical uppercase `T`/`Z` whole-second Trust VC timestamps.
- **Tests:** `packages/wot-core/tests/ProtocolInterop.test.ts`, `packages/wot-core/tests/AttestationWorkflow.test.ts`, and `packages/wot-core/tests/VerificationWorkflow.test.ts` cover fractional rejection, offset normalization, `validUntil`/`exp` matching, and canonical Trust VC-JWS creation timestamps.
- **Open questions:** None for this slice; real-life-org/wot-spec#60 normatively resolved fractional Trust VC timestamp handling.
